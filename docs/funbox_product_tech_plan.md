# 闲趣盒子 产品与技术总方案

## 1. 文档概述

### 1.1 项目名称
- 产品名称：`闲趣盒子`
- 代码项目名：`FunBox`

### 1.2 项目目标
`闲趣盒子` 是一个基于 Flutter 构建的离线小游戏合集，面向飞机、游轮、地铁、长途通勤、家庭共享设备等无网或弱网场景。它的目标不是做重运营游戏平台，而是提供一个打开就能玩的纯单机游戏盒子，让用户在一个 App 内稳定体验多种熟悉小游戏。本文中提到的 `FunBox` 特指当前仓库和代码项目名。

### 1.3 产品原则
- 即开即玩：不依赖注册、登录、联网校验或复杂初始化流程。
- 纯净单机：所有核心玩法在本地完整运行，断网不影响进入、游玩、暂停、继续和结算。
- 熟悉规则：优先选择用户心智成熟、学习成本低的经典小游戏。
- 碎片友好：适配 3 到 20 分钟的短时娱乐场景，支持随时中断和继续。
- 平台优先：先建设统一平台能力，再接入不同类型小游戏，避免每个游戏各做一套基础设施。

## 2. 产品定位与目标用户

### 2.1 目标用户
`闲趣盒子` 的首要目标用户是全年龄休闲玩家。重点覆盖以下几类人群：
- 出行用户：在飞机、游轮、高铁、地铁等不稳定网络环境中寻找可随时开启的娱乐内容。
- 家庭用户：一台设备由多年龄段共享，希望规则直观、界面友好、没有复杂门槛。
- 轻度玩家：偏好经典小游戏，不希望面对账号体系、社交绑定、广告打断和复杂养成。

### 2.2 使用场景
- 无网或弱网环境下的娱乐补充。
- 短时碎片时间，如排队、候机、通勤。
- 家庭或朋友间轮流使用同一设备进行单机游玩。
- 不方便开启重度游戏或大型内容时的轻量替代。

### 2.3 核心价值
`闲趣盒子` 的核心价值是把多种熟悉小游戏放进统一、轻量、稳定的离线平台中。用户不需要下载多个独立 App，也不需要担心网络状态，只需打开 `闲趣盒子` 就可以直接进入想玩的游戏，并在中断后继续之前的进度。

### 2.4 首版明确不做
首版不纳入以下方向：
- 联网对战
- 账号体系
- 云存档
- 广告系统
- 重社交功能
- 重养成或重收藏体系
- 复杂货币系统
- 每日签到、在线活动、限时运营玩法

这些能力不符合当前“纯净单机、离线可玩”的核心定位，也会稀释第一阶段的架构目标。

## 3. 产品形态与信息架构

### 3.1 顶层产品结构
`闲趣盒子` 的产品结构围绕“统一大厅 + 标准化游戏流程”展开，顶层至少包含以下页面或功能区：
- 游戏大厅
- 最近游玩 / 继续游戏
- 统一设置
- 规则说明

### 3.2 游戏大厅
游戏大厅是 App 的主要入口，承担以下职责：
- 展示可玩的小游戏列表。
- 按分类或卡片方式组织游戏。
- 为每个游戏提供清晰入口，包括“开始游戏”“继续游戏”“查看规则”。
- 展示最近游玩的局部状态，帮助用户快速回到中断的游戏。

建议大厅默认优先展示：
- 最近游玩
- 首批推荐游戏
- 按类型分类的全部游戏

### 3.3 游戏统一骨架
所有小游戏都应尽量复用一致的用户流程和页面骨架，至少包括：
- 开始页：展示游戏简介、模式、难度、是否继续上局。
- 对局页：实际游玩界面。
- 暂停层：提供继续、重开、查看规则、返回大厅。
- 结算页：展示胜负、得分、用时等结果，并提供再来一局或返回大厅。

### 3.4 一致的用户路径
每个小游戏默认遵循同一条路径：

`大厅进入 -> 选择模式或难度 -> 开始游玩 -> 暂停或切出 -> 继续游戏 -> 结算 -> 再来一局或返回大厅`

这个统一路径能减少用户的学习成本，也能约束不同游戏接入平台时的页面和状态设计。

## 4. 技术架构与模块边界

### 4.1 架构分层
`FunBox` 代码架构采用三层结构：
- 平台层
- 游戏接入层
- 游戏实现层

### 4.2 平台层职责
平台层负责所有跨游戏复用的通用能力，是整个项目的稳定基础。职责包括：
- 应用路由与页面导航
- 主题与基础视觉规范
- 音效与震动等全局开关
- 本地存档与恢复入口
- 统一设置页
- 统一页面壳和通用弹层
- 游戏注册中心
- 最近游玩数据汇总
- 规则说明内容入口

平台层不实现具体游戏规则，也不直接管理某个游戏的独有对局逻辑。

### 4.3 游戏接入层职责
游戏接入层负责约束小游戏如何接入平台，保证不同玩法都能在统一框架下运行。职责包括：
- 定义游戏元数据格式
- 约束游戏入口注册方式
- 约束局内生命周期事件
- 统一结算数据结构
- 统一存档读写接口
- 约束全局设置和单游戏设置的边界
- 约束单游戏代码目录与资源目录的归属

这一层是平台和具体游戏之间的契约层，目的是降低耦合，避免每增加一个游戏就改动平台底层逻辑。

### 4.4 游戏实现层职责
游戏实现层只负责具体玩法本身，包括：
- 规则引擎
- AI 策略
- 棋盘或牌桌 UI
- 动画与反馈
- 对局状态管理
- 游戏内特殊交互
- 游戏自有图片、图标、插画、音频等资源

具体游戏应尽量保持独立，避免出现斗地主依赖五子棋模块、扫雷依赖贪吃蛇逻辑这类横向耦合。

### 4.4.1 单游戏目录与资源归属原则
每个小游戏都必须在自己的独立目录中维护代码和资源，不与其他游戏共享私有文件。推荐理解方式如下：
- 每个小游戏拥有自己的代码目录。
- 每个小游戏拥有自己的资源目录。
- 游戏封面、卡面、棋子贴图、局内背景、规则插图、音效等都归该游戏目录管理。
- 主框架只保留平台级公共资源，不存放某个小游戏专属图片或音频。

平台级公共资源包括：
- 应用 Logo
- 底部导航图标
- 通用按钮样式资源
- 统一背景纹理
- 平台级空状态插图

不属于平台级公共资源的内容，不应放入主框架资源目录中。

### 4.5 公共接口概念
以下接口是后续实现时应统一理解的核心概念，本方案只定义职责和字段方向，不绑定最终代码形式。

#### `GameDescriptor`
用于描述小游戏本身的注册信息，建议至少包含：
- `id`：唯一标识
- `title`：展示名称
- `category`：游戏分类，如棋类、牌类、益智类、动作类
- `supportsResume`：是否支持继续游戏
- `supportedModes`：支持的模式集合，如人机、双人本机、闯关、无限模式
- `coverAsset`：大厅展示使用的封面图或主视觉资源路径
- `iconAsset`：大厅、收藏、快捷入口可使用的图标资源路径

职责是让大厅、分类、规则页、注册中心都能基于统一元数据工作。

#### `GameSession`
用于描述一局可恢复的游戏状态，要求可序列化。它的职责包括：
- 支持中断后继续游戏
- 支持在大厅显示“继续游戏”入口
- 为不同游戏提供标准化的本地存档容器

不同游戏内部字段可以不同，但平台需要统一理解其“存在、读取、恢复、删除”的生命周期。

#### `GameResult`
用于统一一局结束后的输出结构，供结算页和未来统计能力复用。建议字段方向包括：
- 胜负状态
- 得分或局分
- 用时
- 模式信息
- 难度信息
- 是否由 AI 对战产生

它的目标不是覆盖所有玩法细节，而是提供稳定的跨游戏结算抽象。

#### `GameEntry` 或 `GameModule`
用于表示一个小游戏接入平台的注册入口。职责包括：
- 暴露 `GameDescriptor`
- 提供游戏首页或入口构建能力
- 连接游戏内部状态与平台生命周期
- 提供规则说明和可恢复会话入口
- 暴露该游戏自有资源的访问入口或资源映射

这个概念是未来新增游戏时最重要的扩展点之一。

#### `AppSettings` 与 `PerGameSettings`
设置系统需要分层：
- `AppSettings` 负责全局设置，如主题、音效、震动、通用辅助项。
- `PerGameSettings` 负责单个游戏特有的配置，如五子棋默认先手、斗地主提示强度等。

全局设置不应被某个游戏独占，单游戏设置也不应污染平台层结构。

### 4.6 技术方案原则
状态管理、本地存储、AI 组织方式在当前阶段只给出方向，不锁定具体库，后续再独立出技术选型文档。推荐方向如下：
- 状态管理：选择适合 Flutter 中型应用的响应式状态方案，要求便于平台层与单游戏模块隔离。
- 本地存储：优先使用稳定、轻量、适合结构化小体量数据的本地方案。
- AI 设计：将 AI 与 UI 解耦，避免把决策逻辑写入页面层。
- 模块组织：优先按职责拆分，不按“所有页面一层、所有状态一层”的粗粒度方式堆叠。
- 资源组织：平台层只依赖游戏公开出来的资源描述，不直接接管单游戏资源文件。

## 5. 首批样板游戏规划

### 5.1 样板选择原则
首批样板游戏需要覆盖差异明显的玩法模型，从而帮助平台沉淀更稳固的公共能力。`五子棋` 和 `斗地主` 组合具备以下价值：
- 一款是回合制棋类，状态清晰、规则集中。
- 一款是复杂规则牌类，流程更长、状态更多、AI 更重。
- 两者组合能验证平台对不同复杂度游戏的承载能力。

### 5.2 五子棋
`五子棋` 作为回合制棋类模板，重点验证以下能力：
- 单机 AI 对战
- 先后手切换
- 悔棋
- 重新开始
- 局面存档
- 难度档位

五子棋的意义不在于玩法复杂，而在于它能帮助项目尽早验证：
- 标准回合制对局状态如何建模
- 简洁棋盘界面如何适配统一游戏骨架
- 暂停、继续、结算、重开如何以最低复杂度打通

### 5.3 斗地主
`斗地主` 作为复杂规则牌类模板，重点验证以下能力：
- 发牌流程
- 叫分或抢地主流程
- 出牌校验
- 提示
- 托管
- 本地对手 AI
- 局内状态保存

斗地主是平台里更复杂的样板，它将重点检验：
- 长流程局内状态如何保存与恢复
- 多角色回合如何与统一生命周期接口对齐
- 规则判断、提示系统和 AI 是否能独立于 UI 演进

### 5.4 为什么先做这两款
先做 `五子棋 + 斗地主` 的原因是，它们共同覆盖了两类差异显著的单机玩法：
- 清晰的棋盘式回合博弈
- 复杂的牌类流程与规则校验

当这两类模板稳定后，后续接入贪吃蛇、扫雷、2048、数独等游戏时，可以直接复用大厅、存档、设置、结算、规则说明等平台能力，新增成本会明显降低。

## 6. 版本路线图

### 6.1 V1 平台底座
阶段目标：完成 `FunBox` 代码项目的统一平台能力建设，为接入任何小游戏提供稳定基础。

本阶段聚焦：
- Flutter 项目骨架
- 应用路由
- 统一大厅
- 统一设置
- 本地存档基础能力
- 游戏注册机制
- 继续游戏入口
- 基础规则说明承载方式

V1 的完成标准不是“已经有很多游戏”，而是“新游戏可以按规范接进来，并具备一致的基础体验”。

### 6.2 V2 样板游戏落地
阶段目标：完整实现 `五子棋` 与 `斗地主`，并跑通从大厅到结算的完整闭环。

本阶段聚焦：
- 五子棋完整单机体验
- 斗地主完整单机体验
- AI、存档、暂停、继续、结算在两类游戏中的实际验证
- 平台接口在真实复杂度下的修正和收敛

V2 的完成标准是两款样板游戏都达到“可反复游玩、可中断恢复、体验完整”的水平。

### 6.3 V3 游戏矩阵扩展
阶段目标：在平台与样板稳定后，扩大游戏供给，强化 `闲趣盒子` 作为小游戏合集的价值。

本阶段优先考虑接入：
- 贪吃蛇
- 扫雷
- 2048
- 数独

这类游戏规则成熟、认知成本低、离线体验强，适合作为平台稳定后的快速扩展对象。

## 7. 扩展原则

### 7.1 新游戏接入原则
- 新增小游戏必须复用大厅、设置、结算、存档框架。
- 新增小游戏通过统一注册入口接入，不直接修改平台核心流程。
- 游戏内部逻辑应封装在各自模块中，避免跨游戏依赖。
- 新增小游戏的私有图片、图标、插画、音频等资源必须放在该游戏自己的目录中。
- 大厅、分类、收藏等平台页面展示某个游戏时，应从该游戏模块暴露的资源信息读取素材，而不是从主框架资源目录硬编码引用。

### 7.2 功能优先级原则
后续增强功能应优先围绕离线体验展开，例如：
- 收藏
- 最近玩过
- 主题皮肤
- 轻量统计
- 轻量成就

只有在不破坏纯净单机定位的前提下，才考虑加入这些增强项。

### 7.3 复杂系统引入原则
任何会显著改变产品气质的能力都应单独评估，不默认进入主线。包括：
- 联网系统
- 多账号体系
- 在线排行
- 强运营活动
- 高依赖数值养成

这些方向只有在产品目标发生明显变化时才应重新讨论。

- 拆分大厅与导航设计文档
- 拆分五子棋详细设计文档

## 9. 验收标准

当团队成员阅读本文件后，应能清楚回答以下问题：
- `闲趣盒子` 是什么类型的产品
- 为什么采用平台优先而不是先堆游戏数量
- 首批为什么选择 `五子棋 + 斗地主`
- 平台层、游戏接入层、游戏实现层分别负责什么
- 后续新增小游戏时应遵守什么约束
- 项目会分几个阶段推进，以及每阶段的目标是什么

如果文档不能稳定回答这些问题，就说明仍需补充或收紧表达。
