# 斗地主对局主界面设计文档

## 1. 文档目的

本文档定义 `斗地主` 对局中的主游戏界面布局与视觉层级，用于后续 Flutter 页面实现、组件拆分与交互落地。

设计目标：
- 优先服务“快速看懂该怎么出牌”
- 横屏优先
- 牌桌优先，信息轻量化
- 保持真实产品感，避免大厅式重装饰

## 2. 设计前提

已确认的界面方向：
- 屏幕方向：横屏为主
- 设计重心：牌桌优先
- 中央出牌展示：仅保留最近一手，不常驻完整历史

本设计遵循 `docs/game-ui-style-requirements.md` 的整体风格要求：
- 轻拟物体积感 + 扁平化克制表达
- 轻松、干净、亲和、离线单机感
- 避免重运营、大面积 Banner、强反光、过度装饰

## 3. 页面目标

对局主界面要在 1 秒内回答用户三个问题：
- 轮到谁出牌
- 我现在能不能出
- 我应该关注哪一块牌面

因此页面的优先级是：

1. 手牌与出牌操作
2. 当前回合状态
3. 最近一手出牌
4. 对手剩余牌数
5. 地主身份提示

## 4. 总体布局

### 4.1 布局结构

横屏采用“三段式牌桌 + 底部操作带”结构：

- 左侧：对手 A 信息区
- 中间：牌桌主视觉区
- 右侧：对手 B 信息区
- 底部：玩家手牌区与操作区

推荐的空间分配：

- 左右对手区：各占约 18% - 22%
- 中央牌桌区：约 56% - 64%
- 底部手牌与按钮区：固定存在，占据底部稳定高度

### 4.2 结构示意

```text
┌─────────────────────────────────────────────────────────┐
│ 顶部轻状态栏                                            │
├──────────────┬──────────────────────┬──────────────────┤
│ 对手 A 区     │      牌桌主视觉区     │     对手 B 区     │
│ 头像/张数     │   最近一手出牌卡片    │  头像/张数       │
│ 地主标记      │   当前回合提示        │  地主标记        │
│ 轻状态文案    │                      │  轻状态文案      │
├─────────────────────────────────────────────────────────┤
│                      玩家手牌区                         │
│         出牌 / 提示 / 不出  固定操作条                  │
└─────────────────────────────────────────────────────────┘
```

## 5. 区块设计

### 5.1 顶部轻状态栏

顶部只保留必要信息，不做完整信息展示。

建议包含：
- 返回按钮
- 当前局数或局内阶段提示
- 可选的设置入口

设计原则：
- 不把顶部做成大标题栏
- 不放复杂规则入口
- 不占用牌桌主视觉高度

### 5.2 左右对手区

左右两侧对手区采用对称的轻量状态卡形式，核心只展示：
- 头像或角色占位
- 剩余手牌张数
- 当前状态，如“等待”“出牌中”“Pass”
- 小型地主徽标

设计原则：
- 不展示完整手牌
- 不展示多层文本说明
- 不与中央牌桌争夺视觉焦点

如果对手正在操作，对应区域可通过轻微高亮或脉冲强调当前轮次，但不做强动画。

### 5.3 中央牌桌主视觉区

这是整页最重要的区域，承担“回合说明”和“最近一手出牌”两项职责。

推荐内容：
- 最近一手出牌卡片
- 当前轮到谁出的提示
- 极简回合状态，如“你要压过上一手”

不建议放入：
- 完整历史出牌列表
- 规则说明文本
- 大块装饰性插画

最近一手出牌卡片应根据牌型自动调整宽度与内部排列：
- 单牌、对子、三张：居中紧凑显示
- 顺子、连对、飞机：横向压缩排布，确保可读
- 炸弹、王炸：用更强一点的视觉权重，但仍保持克制

### 5.4 玩家手牌区

玩家手牌区位于底部，是第二视觉焦点，也是最重要的操作对象。

建议规则：
- 手牌横向展开，略有重叠
- 选中的牌轻微上抬
- 非选中状态保持统一高度
- 手牌长度过长时允许自动压缩间距，但保持可点选

玩家手牌区应保证：
- 一眼能看清自己有哪些牌
- 选牌后能迅速判断能否出牌
- 不因对手信息而被压缩得过小

### 5.5 底部操作区

底部操作区与手牌区相邻固定显示，保证用户不用寻找入口。

建议按钮：
- `出牌`
- `提示`
- `不出`

按钮层级建议：
- `出牌` 为主按钮
- `提示` 为辅助按钮
- `不出` 为弱按钮，但始终可见

交互规则：
- 只有当前轮到玩家且存在合法出牌时，`出牌` 才为主要强调态
- `不出` 在可压制场景中高可见，在无意义场景中弱化
- `提示` 不应打断对局节奏，也不应弹出大面积面板

## 6. 信息层级

页面中的信息按以下顺序组织：

1. 当前回合与操作入口
2. 玩家手牌
3. 最近一手出牌
4. 对手剩余张数
5. 地主身份
6. 轻状态提示

页面不应长期展示：
- 完整出牌历史
- 复杂规则说明
- 大段文本提示
- 多入口营销内容

## 7. 视觉方向

### 7.1 背景

建议使用低饱和、轻渐变的背景，营造“桌面感”：
- 深色或中深色背景基底
- 中央区域略亮，形成视线聚焦
- 边缘有轻微模糊或柔光过渡

避免：
- 大面积高饱和色块
- 强烈霓虹
- 花哨纹理堆叠

### 7.2 卡片与面板

所有信息容器统一为轻圆角卡片：
- 圆角明显但不过分夸张
- 阴影克制、短而柔
- 透明度适中，保留层次但不过度发光

### 7.3 牌面表现

牌面应有轻微体积感，但不做重拟物。

建议：
- 白底或浅底牌面
- 统一牌角与边框
- 选中态通过上移、描边和轻微阴影表达
- 炸弹、王炸可增加少量强调色，但不能喧宾夺主

### 7.4 色彩建议

建议主色方向：
- 深海军蓝
- 靛蓝灰
- 柔和紫影
- 暖米白牌面
- 少量青绿作为辅助强调

禁用倾向：
- 过度金色堆叠
- 强荧光色块
- 大面积高饱和冲突色

## 8. 状态设计

### 8.1 常见状态

对局主界面至少需要覆盖以下状态：
- 开局发牌中
- 当前轮到玩家
- 当前轮到对手
- 玩家有合法出牌
- 玩家无合法出牌
- 有人 Pass
- 胜负结算前的最后一轮
- 对局结束

### 8.2 状态反馈方式

状态反馈优先采用：
- 高亮边框
- 轻微位移
- 小型文字提示
- 手牌微动效

不建议：
- 大弹窗打断
- 整屏闪烁
- 大段说明文本

## 9. 交互原则

- 牌桌始终优先于说明信息
- 用户先看到“谁出、我出什么”，再看到“为什么”
- 所有操作尽量单层完成
- 需要长解释的内容，不放在对局主界面
- 对局主界面不承担设置、规则详解和推广入口职责

## 10. 组件拆分建议

为后续实现，建议将主界面拆分为以下组件：

- `LandlordTableShell`
- `LandlordOpponentPanel`
- `LandlordCenterPlayArea`
- `LandlordHandArea`
- `LandlordActionBar`
- `LandlordRoundHint`
- `LandlordCardStack`

拆分原则：
- 每个组件职责单一
- 左右对手区复用同一结构
- 中央区只负责最近一手和轮次状态
- 底部操作区独立管理按钮可用态

## 11. 响应式规则

横屏优先不代表固定分辨率。

需要兼容的变化：
- 小横屏窗口
- 平板横屏
- 桌面窗口缩放

适配原则：
- 对手区先压缩，中央牌桌区保持最大
- 手牌区优先保证可操作
- 按钮宽度可自适应，但顺序不变
- 最近一手卡片可缩放，不可裁掉关键信息

## 12. 验收标准

当对局主界面完成后，应满足以下检查项：

- 用户 1 秒内能看出当前轮到谁
- 玩家能快速找到出牌、提示、不出
- 中央区域只显示最近一手，不出现冗余历史堆叠
- 横屏下牌桌区域明显大于辅助信息区
- 对手信息不会抢占主视觉
- 手牌始终保持足够可点选性
- 页面整体像真实产品，不像活动页或宣传页
