# FunBox 2048 接入设计文档

日期：2026-06-14

## 1. 文档定位

本文档用于定义 `2048` 在 `FunBox` 中的首版接入方案。目标不是扩展 2048 的玩法深度，而是优先完成一个符合平台规范、可恢复进度、可被大厅消费、可继续扩展的小型标准模块。

本文档聚焦以下问题：

- 2048 首版在 `FunBox` 中承担什么角色
- 2048 如何满足现有 `GameModule` / `GameManifest` 接口
- 2048 的最小页面流、状态流和存档结构如何设计
- 哪些能力进入首版，哪些明确不进入首版

## 2. 设计目标

2048 首版接入方案需要同时满足以下目标：

- 作为 `puzzle` 分类中的标准单人游戏接入大厅
- 支持中断后继续游戏
- 接入成本低，不反向污染平台层代码
- 页面流清晰，适合移动端竖屏快速开局
- 后续如果要增加主题、挑战模式、排行榜扩展，不需要推翻当前架构

本次设计优先保证“平台接入完整”和“单局体验稳定”，不追求复杂运营能力。

## 3. 方案比较

### 方案 平台接入优先的经典版

特征：

- 保持单人 4x4 经典玩法
- 完整覆盖 `GameManifest`、`GameModule`、规则入口、结果摘要、会话恢复
- 页面和数据结构按平台统一规范组织

优点：

- 最符合当前 `FunBox` 阶段目标
- 既能作为真实可玩游戏，也能作为 `puzzle` 类模板模块
- 后续扩展模式时平台层无需重写

缺点：

- 比纯玩法版多出一部分接入设计工作

### 推荐结论

本次采用 **方案 B：平台接入优先的经典版**。

原因如下：

- 用户已明确本次优先完成文档，且目标偏“平台接入版”
- `FunBox` 当前已经存在统一大厅与游戏模块规范，2048 应优先对齐这些约束
- 2048 本身玩法成熟，没必要首版先做复杂创新，先把它做成标准模块更有复用价值

## 4. 设计结论

2048 首版定义为：

- 单人
- 经典 4x4 盘面
- 触屏滑动或键盘方向输入
- 支持继续上局
- 支持重开
- 不支持撤销
- 不支持每日挑战
- 不支持联网能力
- 不支持排行榜
- 不支持多模式切换

`supportedModes` 首版保持：

```yaml
supportedModes:
  - classic
```

## 5. 模块定位

2048 在 `FunBox` 中属于标准 `puzzle` 类单局分数型游戏。

它和五子棋、斗地主不同，不依赖回合角色系统，也不依赖复杂 AI；它更适合作为以下能力的模板样例：

- 单人即时操作游戏接入
- 高分驱动型结算
- 小体量状态存档
- 连续局快速重开
- 大厅“继续游戏”入口承接

因此，2048 的价值不只是补一个游戏数量，更是为后续扫雷、数独等 `puzzle` 类模块提供可参考的接入范式。

2048 模块遵循现有游戏模块规范，首版目录建议如下：

│  │  └─ game_result.dart

说明：

## 7. 平台接入设计

### 7.1 Manifest

当前已有 `game_manifest.yaml` 已基本符合首版目标：

```yaml
id: 2048
title: 2048
category: puzzle
iconAsset: games/2048/assets/images/icon.png
coverAsset: games/2048/assets/images/icon.png
supportsResume: true
supportedModes:
  - classic
sortOrder: 10
enabled: true
```

首版保持这个方向，不额外引入新字段。

### 7.2 GameModule

2048 需要实现现有接口：

- `manifest`
- `buildStartPage`
- `buildRuleEntry`
- `createSessionSerializer`
- `createResultAdapter`

建议职责如下：

#### `manifest`

直接暴露 2048 的静态元数据，供大厅、分类、收藏和最近游戏消费。

#### `buildStartPage`

返回 2048 开始页，负责：

- 展示游戏标题与简述
- 展示最高分或最近分数信息
- 如果存在可恢复局面，则显示 `继续游戏`
- 始终显示 `开始新局`
- 提供 `查看规则` 入口

#### `buildRuleEntry`

返回规则入口页或规则卡片，内容包括：

- 基本目标：合成 2048
- 基本操作：上下左右滑动
- 结束条件：盘面无空位且无法继续合并
- 分数来源：每次合并产生的数值累计到总分

#### `createSessionSerializer`

返回 2048 专属 `GameSessionAdapter`，负责把运行中局面编码为平台可存储数据。

#### `createResultAdapter`

返回 2048 专属 `GameResultAdapter`，负责将结算结果转成跨游戏通用摘要。

## 8. 页面流设计

2048 首版页面流保持最短路径：

```text
大厅进入
→ 2048 开始页
→ 开始新局 / 继续游戏
→ 2048 对局页
→ 暂停或结束
→ 再来一局 / 返回大厅
```

### 8.1 开始页

开始页承接平台统一入口，不直接跳过。

开始页展示内容建议包括：

- 游戏图标
- 游戏标题 `2048`
- 一句简介：滑动数字方块，合成更大的数字
- 主按钮：`开始新局`
- 条件按钮：`继续游戏`
- 次按钮：`查看规则`

如果存在存档：

- 优先展示 `继续游戏`
- 同时保留 `开始新局`
- 文案上可附带当前分数和最大数字概览

### 8.2 对局页

对局页是核心页面，结构建议自上而下为：

- 顶部栏：返回、重开、规则入口
- 分数区：当前分数、历史最高分
- 盘面区：4x4 网格
- 底部提示区：操作提示或结束提示

对局页交互要求：

- 接收上/下/左/右四向输入
- 只有在局面发生变化时，才生成新数字
- 每次有效移动后检查是否达成 2048 和是否游戏结束
- 应支持应用切后台后恢复

### 8.3 暂停与返回

2048 不需要独立复杂暂停页，但需要统一的离开保护策略：

- 用户返回大厅前，如果当前局面仍可继续，先自动保存
- 自动保存成功后允许退出
- 不强制二次确认

### 8.4 结算状态

2048 没有传统“胜利后立刻强制收尾”的必要，因此建议：

- 达成 2048 时弹出轻量祝贺提示，但允许继续挑战更高分
- 真正结算时机以“无合法移动”为主
- 游戏结束后展示结果面板：本局分数、最大数字、是否达到 2048、再来一局

这更符合用户对 2048 的经典预期。

## 9. 状态与规则设计

### 9.1 核心运行状态

单局至少包含以下状态：

- `board`: 16 个格子的数字状态
- `score`: 当前分数
- `bestTile`: 当前局面中的最大数字
- `hasReached2048`: 是否已到达过 2048
- `isGameOver`: 是否无合法移动
- `moveCount`: 有效移动次数
- `startedAt`: 本局开始时间
- `lastPlayedAt`: 最近一次有效操作时间

### 9.2 盘面表示

推荐使用固定长度 16 的一维数组表示 4x4 盘面。

原因：

- 序列化简单
- 存档体积小
- 逻辑层易于统一做行列映射

示例：

```text
[
  0, 2, 0, 4,
  0, 8, 8, 0,
  2, 0, 0, 0,
  0, 0, 0, 0
]
```

其中 `0` 表示空格。

### 9.3 移动规则

每次输入方向后按以下顺序处理：

1. 提取该方向对应的 4 条线
2. 去除空格
3. 相邻相同数字只合并一次
4. 合并结果回填
5. 若盘面发生变化，则随机生成一个新数字
6. 更新分数、最大数字、移动次数
7. 检查是否结束

新数字生成规则保持经典：

- 默认生成 `2`
- 小概率生成 `4`

### 9.4 结束判定

满足以下两个条件时游戏结束：

- 盘面已无空格
- 任意方向都无法产生新变化

### 9.5 达成 2048

首版将“达到 2048”视为里程碑，而不是强制终局条件。

理由：

- 更符合经典 2048 习惯
- 更利于高分型结算
- 不影响平台层结果摘要

## 10. 存档设计

### 10.1 是否支持恢复

首版明确支持恢复，因此 `supportsResume` 保持 `true`。

### 10.2 会话数据结构

2048 的 `GameSession` 建议至少包含：

- `version`
- `mode`
- `board`
- `score`
- `bestTile`
- `hasReached2048`
- `moveCount`
- `startedAt`
- `lastPlayedAt`
- `randomSeed` 或等效随机状态信息

说明：

- `version` 用于后续会话结构升级
- `mode` 首版固定为 `classic`，但保留字段有利于以后扩展
- `randomSeed` 是否必须持久化，取决于实现方式

如果随机生成逻辑是“仅依赖当前状态，不要求回放一致”，则可不保存随机种子；如果未来要支持严格复盘，则应保留。

### 10.3 自动保存策略

推荐自动保存策略：

- 每次有效移动后保存一次
- 应用进入后台时保存一次
- 返回大厅前保存一次

这样能最大化保证“继续游戏”入口稳定可用。

### 10.4 失效策略

以下情况应清理当前会话：

- 用户点击 `开始新局` 并确认覆盖当前进度
- 游戏已结束且用户选择 `再来一局`

不建议在“达成 2048”时自动清档，因为用户可能继续冲高分。

## 11. 结果摘要设计

2048 的 `GameResult` 建议至少包含：

- `score`
- `bestTile`
- `moveCount`
- `durationMs`
- `reached2048`
- `isGameOver`

通过 `GameResultAdapter.toSummary()` 输出给平台的摘要建议为：

```text
{
  "score": 12340,
  "bestTile": 512,
  "moveCount": 182,
  "durationMs": 540000,
  "mode": "classic",
  "reached2048": false
}
```

平台层不需要理解 2048 的全部内部细节，只需要消费这些稳定摘要字段，用于：

- 结果页展示
- 后续轻量统计扩展
- 最近游戏状态说明

## 12. UI 设计方向

2048 属于规则熟悉度高的游戏，首版视觉建议遵循：

- 简洁
- 清晰
- 高对比
- 少装饰
- 快速读盘

建议不要把 2048 做成高度拟物或重插画页面，因为这会削弱盘面可读性。

### 12.1 盘面视觉原则

- 空格与数字块层级清楚
- 数字越大，颜色差异越明显
- 动效以滑动和合并反馈为主
- 不做复杂粒子特效

### 12.2 页面视觉优先级

从高到低应为：

- 盘面
- 当前分数
- 主操作按钮
- 规则与次级入口

### 12.3 平台一致性

2048 开始页和规则页应跟随 `FunBox` 平台风格，但对局页允许更强的游戏内聚焦感。也就是说：

- 开始页更像平台页面
- 对局页更像游戏页面

这种切分更符合现有平台架构。

## 13. 测试重点

2048 首版至少要覆盖以下验证点：

- 向四个方向移动时，压缩和合并规则正确
- 单次移动中同一数字不重复连锁合并
- 只有有效移动后才生成新数字
- 分数累计正确
- 达成 2048 时只提示、不强制结束
- 无合法移动时正确结束
- 会话编码与解码前后局面一致
- 恢复后可继续正常移动
- `GameResultAdapter` 输出字段稳定

## 14. 非目标范围

以下内容明确不进入本次首版设计：

- 撤销一步
- 多盘面尺寸
- 每日挑战
- 排行榜
- 成就系统
- 云存档
- 联网能力
- 皮肤系统
- 广告与商业化入口

这些能力都可以作为后续独立专题，而不应挤进当前平台接入版。

## 15. 实施建议

建议将 2048 作为 `puzzle` 类模板模块推进，实施顺序如下：

1. 先完成 `logic/` 中的纯规则引擎与测试
2. 再实现 `GameSession` / `GameResult` 结构
3. 然后接上 `GameModule` 入口和开始页
4. 最后完成对局 UI 和恢复流程验证

这样可以优先锁定最容易出错的规则与存档边界，再补 UI。

## 16. 验收标准

当 2048 首版满足以下条件时，视为符合本设计文档：

- 存在合法的 `game_manifest.yaml`
- 能在大厅和分类页正常显示
- 能从开始页进入新局
- 有存档时能显示 `继续游戏`
- 退出后能恢复到最近有效局面
- 游戏结束后能展示分数和最大数字
- 不需要修改大厅专有逻辑即可完成接入
- 代码与资源保持在 `code/games/2048/` 自身目录内

## 17. 结论

2048 在 `FunBox` 中的首版不应被设计成“玩法创新项目”，而应被设计成一个标准、稳定、可恢复、可复用的 `puzzle` 模块。

本次推荐方案的核心不是“把 2048 做复杂”，而是“把 2048 做规范”。只要接入边界、会话恢复和结果摘要设计正确，它就能同时承担真实可玩游戏和平台模板样例两种角色。
