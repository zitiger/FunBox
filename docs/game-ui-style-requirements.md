# 闲趣盒子游戏界面风格设计要求

## 1. 文档目的

本文档用于统一 `闲趣盒子` 的游戏大厅与平台级页面的视觉风格要求，作为后续页面设计、切图、图标重绘、Flutter 还原和资源接入的执行标准。

本文档聚焦以下内容：

- 整体视觉气质
- 页面结构倾向
- 游戏图标与插画规范
- 哪些元素应切图，哪些元素必须代码绘制
- 资源归属与命名要求

## 2. 总体风格结论

当前确认的整体方向为：

- `轻拟物体积感 + 扁平化克制表达`

具体含义如下：

- 保留轻拟物风格中的层次、材质感和亲和力
- 保留少量柔和高光、圆角、浅阴影和轻微立体结构
- 避免厚重拟物、强反光、强发光和过度装饰
- 借鉴扁平化设计的克制表达，让界面更整洁、更稳定
- 让页面更像真实可落地产品，而不是概念宣传图或活动海报

为避免歧义，后续文档中不再使用 `A`、`B` 这类仅在讨论过程里成立的简称。

## 3. 产品气质要求

整体气质应体现：

- 轻松
- 干净
- 亲和
- 离线单机
- 即开即玩
- 有游戏感，但不重运营

明确避免：

- 重运营手游大厅感
- 内容平台感
- 强活动海报感
- 复杂炫光和噪点堆叠
- 大面积轮播 Banner
- 高频弹窗感和过度装饰

## 4. 色彩与材质要求

推荐主色方向：

- 深海军蓝
- 靛蓝
- 柔和紫色阴影
- 暖橙色强调色
- 奶油白文字
- 少量青绿作为辅助点缀

材质表现要求：

- 卡片可以有轻微体积感
- 图标可以有柔和高光和轻微光泽
- 阴影要克制，不能厚重
- 背景允许有非常轻的渐变和氛围感，但不能喧宾夺主

禁止：

- 重霓虹
- 强镜面反射
- 廉价糖果色堆叠
- 大面积高饱和冲突色

## 5. 布局与信息层级要求

页面结构应优先服务“快速进入游戏”，而不是展示视觉概念。

总体布局原则：

- 信息层级清楚
- 分区明确
- 卡片结构统一
- 主操作位置稳定
- 1 到 2 次点击可进入游戏

页面职责要求：

- 首页：打开就玩，优先承接继续游戏、最近游玩、推荐和分类捷径
- 分类：完整浏览全部游戏
- 收藏：回到用户偏爱的游戏集合
- 设置：承载稳定、低频、全局能力

## 6. 游戏主界面通用规则

本节用于约束各类小游戏的主界面、对局页、规则页、结果态等核心页面，适用于棋盘类、合成类、卡牌类、答题类和其他以“立即开始、持续游玩”为目标的页面。

通用原则如下：

- 主界面应优先服务“快速开始”，不要先做概念展示
- 页面最重要的玩法区域必须拥有最高视觉权重，例如棋盘、卡牌区、题目区或主战场
- 顶部信息只保留必要数据，辅助信息必须低于玩法主体
- 返回按钮应明确可见，适合放在左上角，承担退出或返回入口职责
- 右上角如需放操作入口，优先使用图标按钮或克制的短入口，不要做成大字宣传按钮
- 主操作按钮必须层级清楚，但不能压过核心玩法区域
- 底部只保留轻量引导，不承载大段说明
- 页面中任何会变化的内容，都应由代码渲染，不应写死在切图里
- 结果态应轻量、简短、可继续操作，不要做成活动海报
- 规则页应短、稳、可快速关闭，不应做成长文档页
- 页面整体要像真实上线产品，而不是宣传图、开屏海报或概念稿

如果某个小游戏的玩法核心不是棋盘，也应遵守同样的层级原则：让玩家第一眼看到“怎么开始玩”，而不是“有多少装饰”。

## 7. 已确认的页面风格要求

### 首页

- 保留“大标题 + 主视觉区域 + 最近游戏 + 分类胶囊 + 推荐卡片 + 底部导航”的结构
- 视觉上接近现稿，但必须收成更产品化
- 主视觉区域不能做成活动海报
- 最近游戏与推荐游戏应共用同一套图标体系

### 分类

- 按固定分类组织，不因游戏少而临时打散结构
- 游戏卡片采用规整网格
- 已确认分类页使用 `每行 3 个游戏` 的布局
- 卡片应小巧、模块化、便于复用

### 收藏

- 复用分类卡片思路，但交互更偏“快捷继续/开始”
- 空状态必须完整友好，不能像没做完

### 设置

- 使用清晰的分组卡片
- 更偏真实产品设置页，不带概念海报感
- 即使条目未开放，也应保持结构稳定

## 8. 游戏图标设计要求

游戏图标是跨页面复用的核心视觉资产，要求如下：

- 同一个游戏在首页、分类、收藏中优先复用同一张图标
- 图标主体应清晰、强识别、可缩放
- 图标风格必须统一
- 图标允许有轻拟物质感，但不能太厚重

图标交付要求：

- 输出为透明底 PNG
- 不带外层边框
- 不带固定圆角底板
- 不带容器背景
- 不带标题文字
- 不带百分比
- 不带进度条
- 不带状态文案

这样做的原因是：

- 方便后续放入不同卡片容器
- 方便在首页、分类、收藏中复用
- 避免图标和具体页面布局强耦合

## 9. Banner / 插画类元素要求

首页主视觉若需要插画类素材，应遵守：

- 只切插画主体
- 标题、副标题、按钮、标签由代码绘制
- 插画风格与游戏图标同源，但可以更完整一些
- 不做成活动宣传图

## 10. 代码绘制与切图边界

以下元素建议切图：

- 单个游戏图标
- 首页主视觉插画主体
- 少量确实有明显插画质感的平台级装饰物

以下元素必须代码绘制：

- 页面标题
- 副标题
- 搜索按钮容器
- 分类胶囊
- 卡片底板
- 开始 / 继续 / 重开按钮
- 底部导航栏
- 所有进度条
- 所有百分比
- 所有最近游玩时间
- 所有状态标签
- 设置页列表行和开关容器

原则是：

- 只把“有明显视觉价值的静态插画元素”做成图片
- 所有结构、文案、状态、交互相关内容都交给代码

## 11. 文案与动态信息要求

任何会变化的信息都不应写死在图片中，包括：

- 游戏进度，如 `xx%`
- 最近游玩时间
- 按钮文案
- 分类名称
- 收藏状态
- 设置项当前值

这些内容必须由代码渲染。

## 12. 资源归属要求

资源归属必须与产品技术文档保持一致：

- 每个小游戏的专属资源放在自己的目录中
- 平台页面通过游戏模块公开的资源信息读取这些图片
- 主框架不维护单个游戏专属图标仓库

推荐目录：

- `code/games/<slug>/assets/images/icon.png`

平台公共资源才允许放在平台目录，例如：

- `code/assets/images/home/`
- `code/assets/images/icons/`

平台级入口图标，例如“更多游戏”，不属于单个游戏资源，应放在：

- `code/assets/images/icons/more_games.png`

## 13. 命名与复用原则

命名原则：

- 资源名表达“它是什么”，不要表达“它在哪个页面”
- 优先使用单游戏统一图标，而不是 `recent_*`、`card_*` 这类位置命名

不推荐继续扩张的命名方式：

- `recent_landlord.png`
- `recent_2048.png`
- `card_landlord.png`
- `card_more_games.png`

推荐方向：

- `games/landlord/assets/images/icon.png`
- `games/2048/assets/images/icon.png`
- `assets/images/icons/more_games.png`

## 14. 设计验收标准

后续新增页面、图标或切图资源时，应至少满足以下问题都能回答为“是”：

- 这张图是否符合“轻拟物体积感 + 扁平化克制表达”的风格方向
- 它是否看起来像真实产品，而不是海报
- 同一个游戏是否可以在多个页面复用
- 图片里是否避免写入动态信息
- 该资源是否放在正确的归属目录
- 这项内容是否真的需要切图，而不是应由代码绘制

如果其中任一项答案是否定的，就应先调整后再入库。
